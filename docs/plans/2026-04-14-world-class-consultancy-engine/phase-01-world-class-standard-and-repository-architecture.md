# Phase 01 - World-Class Standard And Repository Architecture

## Objective

Define what "world class" means for this repository, codify it in repo-level instructions, and create the structural rules that every future skill must follow.

## Why This Phase Comes First

The current system is strong but uneven. Some skills are strategically rigorous, some are highly localised, and some are better than others at handling legal, cultural, and commercial nuance. Before adding new skills, the repository needs a formal operating standard that prevents drift.

## Core Outputs

1. A written `world-class consultancy standard` document under `docs/` that defines quality, global adaptability, strategic depth, and commercial credibility.
2. Updates to `README.md`, `AGENTS.md`, and `CLAUDE.md` so market adaptation becomes mandatory rather than optional.
3. A standard skill architecture template for all future retrofits and new skills.
4. A repository-wide quality checklist for strategy skills, platform skills, commercial skills, and reporting skills.

## Repository Work

### 1. Define the world-class benchmark

Create a benchmark document covering:

- strategy quality,
- localisation quality,
- analytical quality,
- client-facing professionalism,
- governance and risk handling,
- AI quality control,
- commercial usefulness.

This document should state clearly that "world class" does not mean "generic global". It means the system can adapt credibly to market, sector, maturity, and risk context while preserving strategic sharpness.

### 2. Standardise mandatory input blocks

Every high-value skill should eventually contain, at minimum:

- market context,
- sector context,
- audience context,
- commercial objective,
- risk/compliance context,
- success metrics,
- explicit assumptions,
- confidence limits.

### 3. Create a shared quality rubric

The rubric should score deliverables on:

- clarity of diagnosis,
- evidence of strategic choice,
- market fit,
- localisation quality,
- measurement credibility,
- implementation practicality,
- professionalism of output,
- ethical and legal awareness.

### 4. Establish retrofit tiers

Categorise the repository into tiers:

- Tier A: skills that directly shape strategy, proposals, pricing, reporting, and platform selection.
- Tier B: skills that shape execution and content operations.
- Tier C: supportive or specialist skills that can be retrofitted later.

This keeps the transformation realistic and prevents low-leverage work from crowding out high-leverage changes.

## Skills And Files Most Affected

- `README.md`
- `AGENTS.md`
- `CLAUDE.md`
- `skill-writing`
- `skill-safety-audit`
- all strategy, platform, proposal, reporting, and AI policy skills in later phases

## Completion Criteria

- The repository has a single documented definition of world-class quality.
- Repo-level instructions explicitly state that East Africa is the fallback, not the hard-coded default for every job.
- A standard input and quality pattern exists for future skill updates.
- The transformation sequence is clear enough that future edits can be audited against it.

## Risks To Control

- Making the standard too abstract to guide actual rewrites.
- Defining quality in a way that rewards length over judgement.
- Introducing global ambition without giving operators a way to express uncertainty or request expert review.

## Reading Material To Buy For This Stage

- *Playing to Win* by A.G. Lafley and Roger L. Martin  
  Use it to sharpen strategic choice and stop "comprehensive but generic" outputs.
- *Good Strategy/Bad Strategy* by Richard Rumelt  
  Use it to make every future strategy skill diagnose the problem before proposing action.
- *Strategy Is Your Words* by Mark Pollard  
  Use it to improve the precision, force, and usefulness of written strategy outputs.
